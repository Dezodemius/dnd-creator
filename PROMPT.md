# Промт для Claude (Sonnet): собрать конвейер генерации персонажей

Ты работаешь в корне репозитория `dnd-creator`. Построй здесь конвейер
«анкета игрока → готовый персонаж в хранилище» целиком, от формы до тестов.
Читай этот файл как техзадание; спорные мелочи решай сам в пользу простоты.

## Контекст

Это публичный репозиторий-хранилище персонажей D&D 5e (редакция 2014)
одного мастера. Язык продукта — русский, весь UI, тексты и коммиты на русском.

Мастер вставляет в веб-форму либо сырую анкету игрока (тогда персонажа надо
придумать), либо уже готовое описание персонажа (тогда только конвертировать).
GitHub Action с Claude обрабатывает вставленное и складывает результат в папку
персонажа. Дальше мастер импортирует JSON в longstoryshort.app (LSS), при
необходимости правит и печатает PDF.

В репо уже лежат два документа — прочитай оба целиком до начала работы:

- `docs/lss_step1_extract_prompt.md` — контракт извлечения: системный промт,
  который превращает текст персонажа в плоский семантический JSON, и
  Pydantic-модель `CharacterDraft`. Там же раздел «Что остаётся на шаг 2» —
  спецификация детерминированного сборщика LSS.
- `docs/Нимуэ_Плут_to_lss.json` — реальный экспорт LSS. Источник истины по
  каждому ключу формата и golden-файл для тестов.

Ключевой принцип из этих документов — не нарушать его нигде: **LLM понимает
смысл, код собирает формат.** Claude никогда не пишет LSS JSON напрямую;
он выдаёт только плоский `CharacterDraft`, а LSS-файл детерминированно
собирает Python.

## Целевая структура персонажа

```
characters/<slug>/
  <slug>.lss.json   # импорт для longstoryshort.app — собирает build_lss.py
  README.md         # человекочитаемый лист персонажа на русском
  portrait.*        # арт — мастер добавляет руками позже
  sheet.pdf         # печатный лист из LSS — мастер добавляет руками позже
```

`slug` — имя персонажа в нижнем регистре, транслитом, через дефис
(«Нимуэ» → `nimue`, «Торин Дубощит» → `torin-duboshchit`). Если папка уже
существует — добавь суффикс `-2`, `-3`, ничего не перезаписывай.

В `README.md` персонажа: имя заголовком, строка «класс · раса · уровень ·
предыстория · мировоззрение», затем `![Портрет](portrait.png)` (пусть битая
ссылка живёт, пока арт не добавлен), затем компактный лист: характеристики
таблицей, КД/хиты/скорость, навыки с владением, атаки, заклинания по уровням,
снаряжение, умения, черты характера и предыстория. В конце — блок «Исходная
анкета» с текстом из формы под спойлером `<details>`.

## Что построить

### 1. Форма — `index.html` в корне

Один самодостаточный файл: HTML + CSS + vanilla JS, никаких фреймворков,
сборки и внешних CDN. Тёмная аккуратная тема, всё по-русски. Публикуется
через GitHub Pages (main, root).

Поля:

- Режим: радио «Анкета игрока» (Claude придумывает персонажа) /
  «Готовый персонаж» (только конвертация). По умолчанию — анкета.
- Имя персонажа (опционально — из анкеты Claude возьмёт или придумает сам).
- Уровень (число 1–20, по умолчанию 3).
- Большая textarea «Текст анкеты / персонажа» (обязательное).
- Textarea «Указания мастера» (опционально: сеттинг, запреты, пожелания).
- Поле «GitHub token» + кнопка «запомнить»: fine-grained PAT хранится
  **только в localStorage**, никогда не попадает в репозиторий и запросы
  куда-либо кроме api.github.com.

Сабмит через GitHub Contents API (`PUT /repos/{owner}/{repo}/contents/...`):
создаёт файл `inbox/<ГГГГММДД-ЧЧММСС>-<slug-или-anketa>.md` с
YAML-frontmatter (`mode`, `level`, `name`, `master_notes`) и текстом анкеты
в теле. Сообщение коммита: `claude: анкета <имя или дата>`. owner/repo захардкодь
по origin этого репозитория. После успеха покажи ссылку на вкладку Actions
и очисти textarea; при ошибке — человекочитаемое сообщение (401 → «проверь
токен», 404 → «токену не выдан доступ к репо» и т.п.).

### 2. Конвейер — `.github/workflows/generate.yml`

- Триггер: `push` в `main` с фильтром `paths: [inbox/**]` плюс
  `workflow_dispatch` с input'ом «путь к файлу в inbox».
- **Защита от зацикливания обязательна**: результат конвейера — тоже push
  (он удаляет файл из inbox), поэтому джоба должна пропускаться, если коммит
  сделан ботом: `if: github.actor != 'github-actions[bot]'` (и коммить
  результат от имени `github-actions[bot]`).
- `permissions: contents: write, id-token: write`.
- `concurrency: generate-${{ github.ref }}` без отмены — очередь, а не гонка.
- Шаги: checkout (fetch-depth: 2) → определить обрабатываемые файлы
  (`git diff --name-only HEAD^ HEAD -- inbox/` для push, input для dispatch;
  файлов может быть несколько — обработать каждый) → Python 3.12 +
  `pip install -r requirements.txt` → `anthropics/claude-code-action@v1` с
  `claude_code_oauth_token: ${{ secrets.CLAUDE_CODE_OAUTH_TOKEN }}` и
  промтом: «Обработай файлы <список> по инструкции `.github/prompts/generate.md`».
  Разреши инструменты через `claude_args` (Bash, Read, Write, Edit, Glob, Grep),
  режим без интерактивных подтверждений.
- Перед финальным push — `git pull --rebase`, чтобы параллельная вставка
  анкеты не роняла джобу.

### 3. Инструкция конвейера — `.github/prompts/generate.md`

Пошаговый регламент для Claude внутри Action:

1. Прочитай файл из inbox: frontmatter (режим, уровень, имя, указания) и текст.
2. Если режим «анкета» — создай персонажа по скиллу
   `.claude/skills/character-generator`. Если «готовый персонаж» — пропусти
   генерацию, работай с текстом как есть.
3. Извлеки плоский JSON **строго по системному промту из
   `docs/lss_step1_extract_prompt.md`** и сохрани во временный `draft.json`.
4. Запусти `python scripts/build_lss.py draft.json -o characters/<slug>/<slug>.lss.json`.
   Скрипт валидирует draft по Pydantic и собирает LSS. Если валидация упала —
   прочитай ошибку, исправь `draft.json`, повтори (максимум 3 попытки, дальше
   — честно упасть с внятным сообщением в лог).
5. Напиши `characters/<slug>/README.md` — лист персонажа по шаблону из
   раздела «Целевая структура» (данные бери из draft.json, исходную анкету —
   из inbox-файла).
6. Удали обработанный файл из inbox.
7. Закоммить всё одним коммитом `claude: персонаж <Имя> (<класс> <уровень>)`
   и запушь.

### 4. Скилл — `.claude/skills/character-generator/SKILL.md`

Скилл генерации персонажа 5e (2014) из анкеты игрока. Требования:

- Статы — point buy 27 очков (если в указаниях мастера не сказано иное),
  расовые бонусы поверх.
- Класс, раса, предыстория — из пожеланий игрока в анкете; где игрок молчит,
  выбирай то, что лучше поддерживает описанный образ, и в одном предложении
  фиксируй почему.
- Уровень — из frontmatter; хиты — максимум на 1-м уровне, среднее дальше.
- Стартовое снаряжение класса + предыстории, деньги по правилам.
- Заклинателям — заговоры и заклинания с русскими названиями, английскими
  в скобках и ссылками на dnd.su (как в эталоне Нимуэ).
- Черты характера, идеал, привязанность, слабость — от предыстории (можно
  адаптировать под анкету), плюс связная предыстория на 5–10 предложений
  по мотивам ответов игрока.
- Выход скилла — цельный текст персонажа на русском (не JSON!), со всеми
  числами и владениями, пригодный для шага извлечения.

### 5. Сборщик — `scripts/build_lss.py` + тесты

- Pydantic-модели скопируй из приложения `docs/lss_step1_extract_prompt.md`
  без изменений.
- `build_lss(draft: CharacterDraft) -> dict` реализует всё из раздела
  «Что остаётся на шаг 2»: модификаторы `(score - 10) // 2`, бонус мастерства
  по уровню, СЛ и бонус атаки заклинаний, раскладка 18 навыков и 6 спасбросков,
  обёртки `{"value": ...}`, HTML-блоки (`traits`, `prof`, `background`,
  `notes-1…6`, …), ProseMirror-документы (`attacks`, `equipment`, `features`,
  `spells-level-0…5`), сериализация внутреннего листа в строку поля `data`,
  внешняя оболочка с `edition: "2014"`, `jsonType: "character"`, `version: "2"`.
  Структуру каждого ключа сверяй с эталоном Нимуэ — он источник истины,
  а не твоя память о формате.
- CLI: `python scripts/build_lss.py draft.json -o out.json`; ошибки валидации
  Pydantic печатать в понятном виде (LLM будет их читать и чинить draft).
- **Golden-тест — Definition of Done.** Составь руками
  `tests/fixtures/nimue_draft.json` — `CharacterDraft` Нимуэ, восстановленный
  глазами из эталонного LSS-файла. Тест собирает из него LSS и сравнивает с
  `docs/Нимуэ_Плут_to_lss.json`: распарсить внешний JSON **и** вложенную
  строку `data` в объекты и сравнивать объекты, а не строки; поля, которые
  заведомо не воспроизводятся (`hiddenName`, `avatar`, `createdAt`, `_id`,
  косметические `size` у текстовых блоков), исключи из сравнения явным
  списком с комментарием. Пока golden-тест красный — работа не готова.
- Плюс юнит-тесты на модификаторы, бонус мастерства и раскладку навыков.
- `requirements.txt`: pydantic, pytest. Стиль — ruff-чистый.

### 6. CI тестов — `.github/workflows/test.yml`

pytest + ruff на push/PR в `main` (кроме путей `characters/**`, `inbox/**`).

### 7. `README.md` репозитория

По-русски: что это и как устроен конвейер (схема в 4 строки), как выпустить
fine-grained PAT (только этот репозиторий, permission «Contents: Read and
write»), как положить `CLAUDE_CODE_OAUTH_TOKEN` в secrets, как включить
Pages (main / root), как пользоваться формой, что делать при упавшем ране
(workflow_dispatch с путём к файлу), как добавить арт и PDF в папку персонажа.

## Правила

- Файлы в `docs/` не менять.
- Прямые коммиты в `main`, каждый коммит с префиксом `claude: `.
- Никаких секретов в коде и коммитах; PAT живёт только в localStorage.
- Перед завершением прогони: `pytest` (golden-тест зелёный), `ruff check .`,
  `python scripts/build_lss.py tests/fixtures/nimue_draft.json -o /tmp/x.json`
  отрабатывает без ошибок, `index.html` открывается локально и валидно
  формирует запрос (можно проверить логикой dry-run без токена).
